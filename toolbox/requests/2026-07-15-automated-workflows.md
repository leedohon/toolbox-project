# 자동 작업 처리 워크플로 도입

## 요청

- 어느 작업창에서든 `자동작업처리` 요청을 받으면 실행할 워크플로를 확인하고 선택된 작업을 끝까지 수행한다.
- 구현 결과는 GitHub 반영과 Blogger 게시글 생성 또는 수정까지 포함한다.
- 실행 결과는 JSON으로 저장하고, 새 작업 시작 시 미확인 결과를 보고한다.
- 사용자가 확인했다고 명시한 결과만 `confirm: Y`로 변경하며 이후에는 다시 보고하지 않는다.
- 워크플로는 `데일리 라이트`, `하드`, `올인`, `크리에이트` 네 종류로 운영한다.

## 확정된 구현 방향

- 전체 기준은 저장소 루트 `AUTOMATED-WORKFLOW.md`에서 관리한다.
- 루트와 Toolbox `AGENTS.md`에서 해당 문서를 필수 운영 기준으로 연결한다.
- 실행 결과는 `toolbox/automation-results/<실행시각>-<워크플로>.json`에 실행별로 저장한다.
- 예약 실행은 포함하지 않으며 사용자의 수동 지시를 트리거로 사용한다.

## 워크플로 수량 변경

- 데일리 라이트는 개선 3건, 신규 페이지 개발·적용 또는 통폐합 1건, 추가 아이디어 3건, 공통 모듈·컴포넌트 제안 3건으로 수행한다.
- 하드는 개선 포인트를 10건에서 6건으로 줄이고 공통 모듈·컴포넌트 결함 수정을 1건에서 2건으로 늘린다. 페이지 개발·적용 또는 통폐합 3건 기준은 유지한다.
- 올인은 개선과 신규 페이지 사이의 고정 수량 없이 에이전트가 우선순위를 정하고, 완료와 검증에 필요한 여유를 제외한 가용 토큰을 전량 사용하는 것을 목표로 한다.
- 크리에이트는 기존 페이지 개선이나 아이디어·공통 컴포넌트 제안 없이 신규 페이지 개발과 실제 적용 5건만 수행한다.

## 워크플로 시작 전 예약 종료

- 워크플로 선택 후 실제 작업을 시작하기 전에 시스템 예약 종료 여부를 항상 묻는다.
- 인터페이스가 지원하면 시간 입력과 `아니오` 선택지를 함께 제공하고, 지원하지 않으면 같은 내용을 일반 채팅 질문으로 받는다.
- 사용자가 입력한 남은 시간 또는 종료 시각을 한국 표준시 기준 예약으로 설정하며, 매 실행마다 새로 확인한다.
- 강제 종료는 사용하지 않고 예약·취소 상태를 실행 결과 JSON에 기록한다.
