# 자동 작업 처리 운영 기준

## 목적

이 문서는 사용자가 수동으로 `자동작업처리`를 요청했을 때 적용할 프로젝트 공통 절차를 정의한다. 예약 실행을 전제로 하지 않으며, 어느 작업창에서 요청하더라도 저장소 루트의 `AGENTS.md`를 통해 이 문서를 먼저 확인한다.

## 트리거와 실행 시작

1. 사용자가 `자동작업처리`, `자동 작업 처리`, `자동업무처리`처럼 이 절차를 호출하면 바로 임의 작업을 선택하지 않는다.
2. `데일리 라이트`, `하드`, `올인`, `크리에이트` 중 어떤 워크플로를 실행할지 묻는다. 사용자가 요청문에서 워크플로를 이미 명확히 지정했다면 다시 묻지 않는다.
3. 실행 전에 `toolbox/automation-results/*.json`을 확인하고 `confirm`이 `Y`가 아닌 완료 기록을 먼저 요약 보고한다.
4. 워크플로를 선택한 뒤 실제 작업을 시작하기 전에 시스템 예약 종료를 사용할지 반드시 묻고 사용자의 답을 기다린다.
5. 미확인 기록 보고는 새 워크플로 실행을 막지 않는다. 다만 새 작업의 범위와 충돌하거나 같은 변경을 반복할 위험이 있으면 먼저 사용자 결정을 받는다.
6. 선택된 워크플로는 분석이나 제안에서 멈추지 않고 이 문서와 프로젝트 공통 규약이 허용하는 범위에서 구현, 게시글 생성 또는 수정, 검증, Blogger 반영, Git 커밋·푸시, 공개 배포 확인까지 수행한다.

## 워크플로 시작 전 예약 종료 확인

1. 워크플로 종류가 확정되면 다음 의미의 입력 요청을 표시한다: `시스템 예약 종료를 설정할까요? 종료까지 남은 시간(예: 90분) 또는 종료 시각(예: 23:30)을 입력하거나 아니오를 선택하세요.`
2. 채팅 인터페이스가 선택형 입력 UI를 지원하면 시간 입력과 `아니오` 선택지를 함께 제공한다. 현재 인터페이스에서 사용자 정의 입력창이나 버튼을 제공할 수 없으면 같은 내용을 일반 채팅 질문으로 표시한다.
3. 사용자가 `아니오`라고 답하면 종료를 예약하지 않고 워크플로를 시작한다.
4. 사용자가 시간을 입력하면 한국 표준시 기준으로 실제 종료 예정 시각을 계산해 알려 주고 Windows 시스템 예약 종료를 설정한 뒤 워크플로를 시작한다.
5. `30분`, `2시간`, `90분 후` 같은 남은 시간과 `23:30`, `오후 11시 30분` 같은 당일 시각을 허용한다. 이미 지난 시각, 해석이 여러 가지인 입력과 지나치게 짧아 작업 완료가 불가능한 시간은 다시 입력받는다.
6. 종료 예약은 사용자가 해당 실행에서 시간을 입력한 경우에만 허용한다. 이전 실행의 선택을 기억해 다음 실행에 자동 적용하지 않는다.
7. 강제 종료 옵션은 사용하지 않는다. 예약 취소가 필요하면 사용자 요청을 받아 취소하고, 예약 및 취소 결과를 채팅과 결과 JSON에 기록한다.
8. 예약 시각이 가까워졌는데 작업이 끝나지 않았다면 결과 JSON과 현재 변경 상태를 우선 보존하고, 가능한 범위에서 검증과 Git 동기화를 마친 뒤 미완료 사항을 `blocked`에 기록한다.

## 공통 실행 원칙

### 기계 판독 실행 설정

- 실행 수량·완료 조건은 `toolbox/automation/workflows/<워크플로>.json`을 먼저 읽고, 이 문서는 설명과 정책 판단이 필요할 때만 참조한다.
- 변경 파일에 필요한 검사만 고를 때는 `toolbox/automation/impact-rules.json`을 사용한다. `always`와 릴리스 시 `releaseAlways` 검사는 생략하지 않는다.
- 실행 결과에는 `toolbox/automation/run-state.schema.json`의 `select → implement → validate → publish → commit → deploy → confirm` 단계 상태를 포함한다. 재개할 때 `completed` 단계는 건너뛰고 최초 미완료 단계부터 이어간다.
- 반복 오류와 차단은 `toolbox/automation/block-codes.json`의 코드를 사용하고 재시도 가능 여부와 다음 행동을 함께 기록한다.
- 작업을 시작할 때 요청을 `작업 종류`, `대상 도구`, `버전 단계`, `Blogger 게시`, `Git·배포`, `종료 예약` 여섯 필드로 정규화해 결과 JSON에 기록한다. 이미 확정된 값은 중간에 다시 질문하거나 추론하지 않는다.

- 공개 도구 후보는 `outputs/tools.json`과 각 `versions.json`을 기준으로 정한다. `draft`와 `retired` 도구는 기본 표본에서 제외한다.
- 무작위 점검 대상과 선택 기준은 결과 JSON에 기록한다. 가능한 경우 최근 자동 작업에서 점검한 도구를 피해서 순환하고, 대상 수가 부족하면 존재하는 공개 도구를 모두 사용한다.
- 작업 시작 전에 관련 `toolbox/requests/*.md`, `toolbox/tools/*.md`, 최신 버전 이력과 공통 규약을 확인한다.
- 기존 도구를 변경하면 버전 릴리스 절차를 지키고 이전 버전 파일을 수정하지 않는다.
- 게시글 초안과 수정은 `toolbox-post-draft` 스킬과 `scripts/build-tool-posts.mjs`를 우선 사용한다.
- 기능만 변경된 경우 고정 iframe을 갱신하고, 제목·설명·사용법·FAQ가 변경된 경우 기존 Blogger 게시글도 수정한다.
- 신규 게시글은 기존 도구와 목적 또는 입력·결과 구조가 중복되지 않아야 한다. 유사 기능은 신규 페이지보다 기존 도구 통합을 우선 검토한다.
- 초간단 도구 원칙을 유지한다. 옵션과 기능 수를 늘리는 것보다 핵심 작업의 이해도, 정확성, 속도와 오류 복구 경험을 우선한다.
- 삭제, Blogger 휴지통 이동, 공개 글의 초안 전환처럼 되돌리기 어려운 작업은 이 문서만으로 허용하지 않는다. 사용자의 별도 명시가 필요하다.
- 구현이 완료되고 검증을 통과한 변경만 게시한다. 일부만 완료된 도구나 실패한 변경은 게시하지 않고 결과 JSON에 사유를 남긴다.
- 비밀값, OAuth 토큰과 사용자 데이터는 결과 JSON이나 Git 기록에 포함하지 않는다.

## 데일리 라이트 워크플로

작고 안전한 개선을 여러 도구에 분산하는 가벼운 작업이다.

1. 공개 게시글과 도구를 무작위로 점검해 서로 구분되는 개선 포인트 3개를 선정하고 실제로 개선한다.
2. 기능, 본문, 모바일·데스크톱 UX, 접근성, 오류 상태와 공통 규약 준수 여부를 확인한다.
3. 각 개선은 복잡도를 높이지 않아야 한다. 간단한 기능 추가는 허용하지만 한 화면의 목적을 흐리거나 설정 부담을 늘리지 않는다.
4. 신규 페이지 개발 또는 기존 페이지 통폐합 1건을 완료하고 실제 게시글과 배포에 적용한다. 여러 페이지를 하나의 목적에 맞게 통합하는 작업은 대상 페이지 수와 관계없이 1건으로 센다.
5. 기존 기능과 중복되지 않는 추가 아이디어 3개를 수집한다. 각 아이디어에 사용자 문제, 입력과 결과, 기존 도구와의 차이, 브라우저 단독 구현 가능성, 예상 복잡도와 채택 여부를 기록한다. 이 단계의 아이디어는 제안으로만 남기고 4번의 개발 대상과 중복해 세지 않는다.
6. 공통 모듈 또는 공통 컴포넌트 개선 제안 3개를 작성한다. 영향 범위와 기대 효과를 포함하되, 해당 실행에서 구현하기로 판단한 경우에만 실제 변경한다.

## 하드 워크플로

도구 구성을 적극적으로 정리하고 완성된 신규 결과를 만드는 집중 작업이다.

1. 공개 게시글과 도구를 무작위로 점검해 서로 구분되는 개선 포인트 6개를 선정한다.
2. 선정한 6개 개선을 실제로 적용한다. 기능, 본문, UX와 공통 규약을 함께 점검하며, 단순성을 더 높일 수 있다면 기능의 과감한 제거, 교체 또는 추가를 허용한다.
3. 아이디어 제안만 남기지 않고 총 3건 정도의 페이지 작업을 개발하고 적용한다.
   - 신규 페이지 1개는 1건으로 센다.
   - 여러 페이지의 통합 또는 분리는 대상 페이지 수와 관계없이 하나의 통폐합 작업이면 1건으로 센다.
   - 예: 통폐합 1건과 신규 페이지 2개는 총 3건이다.
4. 공통 모듈과 공통 컴포넌트를 점검해 영향이 큰 결함 또는 이슈 2건을 선정하고 실제로 수정한다.
5. 통폐합된 기존 도구는 `status: retired`와 호환 진입점 규칙을 적용하여 기존 주소가 끊기지 않게 한다.

## 올인 워크플로

한 번의 실행에서 완성 가능한 개선과 신규 페이지 수를 최대화하는 대규모 작업이다.

1. 개선과 신규 페이지의 고정 수량을 두지 않는다. 에이전트가 현재 프로젝트에서 효과가 큰 작업을 판단해 두 유형의 비중과 순서를 정한다.
2. 가용 토큰을 전량 사용하는 것을 목표로 개선 또는 신규 페이지 개발을 계속한다. 다만 마지막 작업의 구현, 검증, 결과 JSON 저장과 완료 보고에 필요한 실행 여유는 반드시 확보한다.
3. 신규 페이지는 구현, 게시글, 버전 기록, 검증과 게시 준비가 모두 완료된 경우에만 게시한다.
4. 미완성 신규 페이지는 공개하거나 카탈로그에 노출하지 않는다. 안전하게 보존할 수 있는 초안 상태로 남기고 결과 JSON에 미완성 사유와 다음 작업을 기록한다.
5. 작업량이 많아도 공통 규약, 기존 주소 호환성, 모바일·데스크톱 검증과 Git·Blogger 배포 확인을 생략하지 않는다.

## 크리에이트 워크플로

신규 페이지 개발과 실제 적용에만 집중하는 제작 전용 작업이다.

1. 기존 공개 도구와 목적 또는 입력·결과 구조가 중복되지 않는 신규 페이지 5개를 개발한다.
2. 아이디어 제안, 기존 페이지 개선과 공통 모듈·컴포넌트 제안은 별도 작업 건수로 수행하지 않는다. 다만 신규 페이지를 완성하는 데 필요한 공통 결함 수정은 허용한다.
3. 각 페이지는 도구 정의, 실행 기능, Blogger 게시글, 버전 이력, 패치노트, 카탈로그 정보와 다국어 UI를 모두 갖춘다.
4. 페이지별 구현과 검증이 완료되는 순서대로 적용하되, 미완성 페이지는 게시하거나 공개 카탈로그에 노출하지 않는다.
5. 신규 페이지 5개의 Blogger 반영, Git 커밋·푸시와 GitHub Pages 배포 확인까지 완료해야 워크플로를 완료로 처리한다.

## 결과 JSON

### 저장 위치와 파일명

- 실행마다 `toolbox/automation-results/<실행시각>-<워크플로>.json` 파일 하나를 만든다.
- 실행시각은 한국 표준시 기준 `YYYYMMDD-HHmmss` 형식을 사용한다.
- 워크플로 식별자는 `daily-light`, `hard`, `all-in`, `create` 중 하나를 사용한다.
- 실행이 일부 실패하거나 차단돼도 결과 JSON을 남긴다.

### 필수 구조

```json
{
  "runId": "20260715-090000-daily-light",
  "workflow": "daily-light",
  "status": "completed",
  "confirm": "N",
  "startedAt": "2026-07-15T09:00:00+09:00",
  "completedAt": "2026-07-15T10:00:00+09:00",
  "confirmedAt": null,
  "summary": "실행 결과 요약",
  "selectedTools": [],
  "changes": [],
  "newIdeas": [],
  "commonModules": [],
  "validation": [],
  "blogger": {
    "created": [],
    "updated": [],
    "skipped": [],
    "status": "completed"
  },
  "git": {
    "branch": "main",
    "commit": null,
    "pushed": false
  },
  "deployment": {
    "status": "pending",
    "urls": []
  },
  "shutdown": {
    "requested": false,
    "scheduled": false,
    "input": null,
    "scheduledFor": null,
    "cancelled": false
  },
  "blocked": [],
  "reportHistory": []
}
```

- `status`는 `completed`, `partial`, `blocked`, `failed` 중 하나다.
- `confirm`은 새 결과를 만들 때 항상 `N`이다.
- 배열 항목에는 대상, 수행 내용, 결과, 관련 파일 또는 공개 URL을 가능한 범위에서 기록한다.
- 제안이 없는 워크플로에서는 해당 배열을 빈 배열로 둔다.

## 미확인 결과 보고와 확인 처리

1. 이 저장소에서 새 작업을 시작할 때 `toolbox/automation-results/*.json`을 확인한다.
2. `confirm`이 `Y`가 아닌 모든 기록을 실행일, 워크플로, 상태, 주요 변경, 게시·Git·배포 상태와 차단 사항 중심으로 먼저 보고한다.
3. 보고할 때 해당 JSON의 `reportHistory`에 보고 시각을 추가한다. 보고했다는 사실만으로 `confirm`을 변경하지 않는다.
4. 사용자가 해당 결과를 확인했다고 명시하면 보고한 실행 ID를 기준으로 해당 JSON의 `confirm`을 `Y`로, `confirmedAt`을 현재 시각으로 갱신한다.
5. 단순히 다음 요청을 보냈거나 대화를 계속했다는 이유로 확인을 추정하지 않는다. 여러 미확인 결과가 있을 때 확인 대상이 불명확하면 실행 ID를 짧게 확인한다.
6. `confirm: Y`인 결과는 이후 새 작업에서 다시 보고하지 않는다.
7. 확인 상태 변경도 저장소에 남아야 하므로 검증 후 Git 커밋·푸시를 시도한다. 권한 또는 네트워크로 실패하면 로컬 JSON은 유지하고 정확한 차단 단계를 보고한다.

## 완료 기준

자동 작업 처리는 다음 조건을 모두 충족해야 완료로 간주한다.

- 선택한 워크플로의 필수 수량과 범위를 처리했거나, 부족한 대상과 이유를 기록했다.
- 변경된 도구의 버전, 게시글, iframe, 카탈로그와 패치노트가 서로 일치한다.
- JSON 구문과 관련 스크립트 검증을 통과했다.
- 변경된 화면을 모바일과 데스크톱에서 확인했다.
- 필요한 Blogger 게시글 생성 또는 수정을 수행하고 공개 상태를 확인했다.
- 관련 변경과 결과 JSON을 커밋하고 `main`에 푸시했으며 GitHub Pages 상태를 확인했다.
- 실패하거나 제외한 항목은 임의 대체하지 않고 결과 JSON에 대상과 이유를 남겼다.
