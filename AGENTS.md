# Blogger 프로젝트 전역 지침

## 자동 작업 처리

- 자동 작업의 선택, 사전 확인, 실행, 결과 보관과 Skill·Markdown 최적화는 루트의 `AUTOMATED-WORKFLOW.md`를 단일 절차로 따른다.

## 프로젝트 구조

1. 상위 프로젝트: `blogger`
   - 블로그 콘텐츠와 독립 실행형 도구를 함께 제공하는 상위 서비스다.
2. 하위 프로젝트 1: `toolbox`
   - 블로그 게시글에 삽입할 수 있는 생성기, 컨버터, 계산기 도구 모음이다.
   - 현재 공개 도구는 QR · 바코드 생성기, JSON 도구, Base64 도구다.
   - 하위 프로젝트의 구체적인 작업 기준은 `toolbox/AGENTS.md`를 따른다.

## Toolbox 프로젝트 정체성

- 프로젝트명: `toolbox`
- 목적: 관리자가 정의한 콘셉트와 규칙을 바탕으로, 사용자가 바로 쓸 수 있는 웹 도구(생성기, 컨버터, 계산기)를 제공한다.
- 초기 우선순위: 블로그나 커뮤니티보다 **도구 탐색과 실행 경험**을 먼저 완성한다.

## 제품 원칙

1. 도구 하나가 독립적인 목적과 명확한 입력·결과를 가진다.
2. 가능한 많은 도구를 설정형으로 운영한다. 관리자는 Markdown 파일로 제목, 설명, 입력값, 실행 규칙을 정의한다.
3. 복잡한 변환, 계산, 파일 처리처럼 안전성과 정확성이 필요한 기능은 검증된 서버 코드로만 실행한다.
4. 각 도구는 검색에 잘 노출될 수 있도록 설명, 사용 방법, FAQ, 관련 도구 영역을 지원한다.
   - 새로 작성하거나 수정하는 Blogger 게시글은 `toolbox/seo-rules.json`의 검색 의도·제목·리드·제목 구조·자연스러운 키워드·FAQ·링크 기준을 따른다.
   - 이 SEO 기준은 게시글 콘텐츠에만 적용하며 `embed/` 실행 기능과 도구 UI에 유입용 문구를 넣지 않는다.
   - 키워드 반복, 숨김 텍스트, 중복 설명과 관련 없는 분량 늘리기는 허용하지 않는다.
5. 처음에는 빠르고 단순하게 만들며, 로그인·댓글·결제 같은 부가 기능은 필요성이 확인된 뒤 추가한다.
6. 각 도구는 아주 간편하고 가볍게 사용하도록 설계한다. 한 화면에서는 해당 도구의 핵심 기능만 충실히 제공하며, 관련 없는 기능·설정·가입 요구를 추가하지 않는다.
7. 파일 입력이 필요한 도구는 파일 선택 버튼과 함께 드래그 앤 드롭 영역을 제공한다.
8. 이미지 영역 편집 도구가 자유 칠하기와 범위 선택을 모두 지원하면, 모드를 명확한 라디오 선택으로 제공하고 드래그 중 적용 범위를 시각적으로 미리 보여 준다. 각 적용은 되돌리기 가능한 한 단계로 기록한다.
8. 모든 도구 페이지는 도구 제목, 한 줄 설명, 핵심 기능 안내를 기본으로 포함한다. 안내는 사용을 방해하지 않게 짧고 분명하게 작성한다.

## 블로거 프로젝트와의 관계

- `toolbox`는 블로거 프로젝트의 하위 구성이다.
- 모든 공개 도구는 Blogger 게시글의 HTML 보기에 그대로 붙여 넣을 수 있는 게시글 HTML과 GitHub Pages에서 실행되는 기능 HTML을 분리해 제공한다.
- Blogger 게시글에는 도구별 고정 주소인 `https://leedohon.github.io/toolbox-project/embed/<도구명>/`을 iframe으로 삽입한다. 게시글을 다시 수정하지 않아도 이 주소의 기능 HTML을 갱신하면 최신 기능이 반영된다.
- `embed/<도구명>/index.html`은 항상 최신 기능을 제공하는 배포 진입점이며, 게시글 제목·설명·사용 방법·FAQ와 사이트 탐색 요소를 포함하지 않는다.
- 게시글 안의 iframe 도구도 독립 페이지와 같은 핵심 기능을 제공하되, 사이트 탐색·부가 설명·중복된 헤더는 표시하지 않는다.
- iframe 기능 HTML은 `postMessage`로 실제 콘텐츠 높이를 부모 게시글에 전달하고, 게시글 HTML은 출처와 도구 식별자를 검증한 뒤 iframe 높이를 갱신해 내부 세로 스크롤이 생기지 않게 한다.
- 블로그 게시글은 HTML 편집기에 그대로 붙여 넣을 수 있는 완성형 HTML 조각으로 출력한다. 기본 구성은 제목, 설명, 기능 안내, 사용 방법, 도구 임베드, FAQ 순서다.

## 초기 도구 범위

- 텍스트 생성기: 제목, 소개문, 해시태그, 문구 등
- 텍스트 컨버터: 대소문자, 줄바꿈, URL 인코딩, JSON 정리 등
- 계산기: 할인율, 날짜, 단위 등

## 콘텐츠 및 도구 정의

- 도구 정의는 `toolbox/tools/*.md`에 둔다.
- 블로그 게시글 원본은 `toolbox/posts/`에 두고, Blogger에 복사할 버전별 HTML은 `outputs/<도구명>/<버전>/`에 둔다.
- GitHub Pages에서 iframe으로 제공할 최신 기능 HTML은 `embed/<도구명>/index.html`에 둔다.
- 버전은 `[major].[minor].[minimum]v` 형식을 사용하며 최초 버전은 `0.0.1v`로 시작한다.
- `minimum`은 간단한 수정, `minor`는 크지 않은 기능 변경, `major`는 기능 개편 수준의 변경에 사용한다.
- 버전을 올릴 때 이전 버전 폴더는 보존하고 새 버전 폴더를 만든다. `minimum` 증가 시 마지막 숫자만 올리고, `minor` 증가 시 `minimum`을 0으로, `major` 증가 시 `minor`와 `minimum`을 0으로 초기화한다.
- 각 버전 폴더에는 완성형 HTML과 해당 버전의 처리 이력을 담은 `patch-notes.json`을 함께 둔다.
- 각 도구 폴더 바로 아래에는 전체 버전 목록과 최신 버전 정보를 담은 `versions.json`을 둔다.
- 각 `versions.json`에는 검색 및 정렬에 사용할 고유한 양의 정수 `index`, 도구 식별자인 `tool`, 표시 제목인 `title`, 카드 설명인 `description`, Blogger 게시글 주소인 `postUrl`, 최신 버전인 `latestVersion`을 포함한다.
- 전체 도구 최신 목록은 `outputs/tools.json`에 두며 각 항목은 `index`, `tool`, `title`, `description`, `postUrl`, `version`을 포함한다. `index`는 중복하지 않고 한번 배정한 값은 바꾸지 않는다.
- `outputs/tools.json`의 `version`은 해당 도구 `versions.json`의 `latestVersion`과 항상 같아야 한다.
- `versions.json` 변경이 `main`에 푸시되면 `.github/workflows/update-tool-catalog.yml`이 카탈로그를 다시 만들고 변경된 결과를 자동 커밋한다.
- 메인 소개글과 공통 화면 문구는 `outputs/site.json`에서 관리한다.
- 새 버전을 만들 때 버전 폴더의 `patch-notes.json`과 도구 폴더의 `versions.json`을 함께 갱신한다.
- 도구 Markdown은 YAML frontmatter에 `title`, `slug`, `type`, `description`, `status`, `inputs`를 포함한다.
- 지원 유형은 우선 `template-generator`, `ai-generator`, `converter`, `calculator`로 제한한다.
- `status`는 기본값을 `draft`로 하며, `published`인 도구만 공개한다.
- 각 공개 도구에는 다음 내용을 제공한다.
  - 제목: 도구가 무엇인지 즉시 알 수 있는 이름
  - 설명: 어떤 입력으로 무엇을 얻는지 알려 주는 한두 문장
  - 기능 안내: 지원 형식, 주요 선택지, 저장 또는 복사 가능 여부 등 핵심 기능 요약
  - 사용 방법: 입력 → 설정 → 실행 순서의 짧은 안내

## 버전 릴리스 작업 절차

1. 작업을 시작하기 전에 대상 도구의 `outputs/<도구명>/versions.json`에서 `latestVersion`과 기존 처리 이력을 확인한다.
2. 사용자 요청을 `minimum`, `minor`, `major` 중 하나로 분류하고 다음 버전을 계산한다.
3. 최신 버전 폴더를 새 버전 폴더로 복사한 뒤 새 버전에서만 HTML을 수정한다. 이미 기록된 이전 버전 파일은 수정하지 않는다.
4. 최신 기능을 `embed/<도구명>/index.html`에 반영하고 게시글 iframe 주소가 이 고정 경로를 사용하는지 확인한다.
5. 새 버전 폴더의 `patch-notes.json`에 해당 버전에서 실제로 처리한 변경만 기록한다.
6. 도구 폴더의 `versions.json`에 새 버전 항목을 추가하고 `latestVersion`을 새 버전으로 갱신한다.
7. `toolbox/post-content.json`에 사용자용 상세 설명과 FAQ를 작성하고 `node scripts/build-tool-posts.mjs <도구명>`을 실행해 상세 설명, FAQ, 누적 패치노트를 갱신한다. 기존 버전 항목은 삭제하지 않는다.
8. `node scripts/build-tool-catalog.mjs`를 실행해 모든 `versions.json`에서 `outputs/tools.json`을 다시 만든다. 카탈로그를 손으로 중복 편집하지 않는다.
9. Blogger 게시글을 발행하거나 주소가 바뀌면 `node scripts/blogger-sync-post-urls.mjs`로 글 제목과 도구 제목을 대조해 `postUrl`과 카탈로그를 갱신한다.
10. `versions.json`에 기록된 HTML 및 패치 노트 상대 경로가 실제 파일을 가리키는지, `outputs/tools.json`과 최신 버전이 일치하는지 확인하고 모든 JSON의 구문을 검증한다.
11. 구현이 끝나면 Blogger 게시글과 iframe 도구를 데스크톱·모바일 화면에서 직접 열어 디자인 이상 여부를 확인한다. 가로 넘침, 비정상적인 폭 축소, 잘림, 중복 스크롤, 간격, 글자 크기, 버튼 줄바꿈, 표·오류 상태를 점검하고 문제가 있으면 수정 후 같은 화면에서 다시 검증한다.
12. 디자인 검증 결과와 실제 수정 사항을 새 버전의 `patch-notes.json`에 기록한다.
13. 사용자에게 새 버전 번호, 변경 등급, 게시글 HTML 및 기능 HTML 경로, 주요 변경 내역과 디자인 검증 결과를 보고한다.

- 모든 공개 게시글은 자주 묻는 질문 영역에 `패치노트` 접이식 항목을 포함한다. 각 버전명과 해당 `patch-notes.json`의 변경 목록을 최신순으로 정적 출력한다.
- 최신 릴리스 HTML에 전체 버전 이력이 누락되면 완료하거나 배포하지 않는다.
- 모든 공개 게시글은 도구별 상세 설명 두 문단 이상과 사용자용 FAQ 네 개 이상을 포함한다. 게시글 수정·배포 방식 같은 운영자용 질문은 공개 본문에서 제외한다.
- 게시글 초안·수정 작업은 설치된 `toolbox-post-draft` 스킬과 `scripts/build-tool-posts.mjs`를 우선 사용해 공통 HTML을 수작업으로 반복 생성하지 않는다.

## Blogger 게시 자동화 원칙

- 전체 운영 절차와 권한 경계는 `toolbox/BLOGGER-OPERATIONS.md`를 따른다.
- 요청에 게시·발행·공개·릴리스가 포함되면 파일 준비에 그치지 않고 Blogger 발행, 게시글 URL 동기화, Git 커밋·푸시, 공개 배포 확인까지 수행한다.
- 요청이 작성·준비·초안 수준이면 Blogger 초안까지만 만들고 공개하지 않는다.
- 실행 기능만 바뀌면 고정 iframe 주소가 최신 기능을 제공하므로 Blogger 본문을 중복 수정하지 않는다.
- 제목, 설명, 사용 방법, FAQ가 바뀌면 새 버전 HTML을 기준으로 기존 Blogger 게시글을 수정한다.
- 테마와 레이아웃은 Blogger API v3 관리 대상이 아니므로 게시글 API 자동화와 분리한다.
- 삭제, 휴지통 이동, 공개 글의 초안 전환은 사용자가 명시적으로 요청한 경우에만 수행한다.

## Git 기록 원칙

- Git 기본 브랜치는 `main`을 사용한다.
- 완성된 작업은 사용자가 달리 요청하지 않는 한 프로젝트 파일 검증 후 관련 변경을 한 커밋으로 저장하고 `main`에 푸시한다.
- 커밋과 푸시를 사용자의 고정 역할로 두지 않는다. 에이전트가 권한을 사용할 수 있으면 직접 처리하고, `.git` 쓰기 또는 원격 네트워크가 차단될 때만 정확한 오류와 최소 사용자 명령을 안내한다.
- 최초 초안 커밋 메시지는 `chore: initialize toolbox draft`를 사용한다.
- 이후 버전 커밋 메시지에는 도구명과 버전을 포함한다. 예: `feat(base64-tools): release 0.1.0v`.
- 커밋 전에 의도하지 않은 파일이나 비밀값이 포함되지 않았는지 확인한다.
- 커밋 성공 후 커밋 해시와 작업 트리 상태를 확인해 사용자에게 보고한다.

## 운영 안전 원칙

- Markdown 안의 지시문은 도구 생성 규칙으로 해석하며, 임의 코드나 셸 명령으로 실행하지 않는다.
- AI API 키와 비밀값은 서버에서만 다룬다.
- 사용자 입력은 유효성 검사와 길이 제한을 거친다.
- 사용자 요청 중 기술적으로 구현할 수 없거나, 안전성·정확성·권한 문제로 신뢰할 수 있게 제공할 수 없는 사항은 작업 범위에서 제외하거나 삭제해도 된다.
- 불가 사항을 제외할 때는 임의의 유사 기능으로 바꾸지 않고, 제외한 대상과 이유를 해당 `requests/*.md` 작업 기록 및 완료 보고에 남긴다.

## 인터페이스 및 디자인 기준

- 전체 분위기는 화이트와 밝은 스카이블루를 기본 톤으로 사용한다. 넓은 여백, 선명한 대비, 과도한 장식 없는 가벼운 화면을 지향한다.
- 기본 동작 버튼은 가로형이며, 읽기 쉬운 스카이블루 배경과 흰 글자, 약 `0.5rem` 수준의 둥근 모서리를 사용한다.
- 공통 색상은 `toolbox/styles/base.css`의 토큰을 기준으로 사용한다. 이후 도구는 이 토큰과 공통 클래스를 우선 재사용한다.
- 취소·되돌리기·보조 동작 버튼은 흰 배경, 검정 글자, 테두리가 있는 가로형 버튼으로 표시한다.
- 크기·수치·강도 등 연속값을 조절하는 기능은 가로 슬라이더를 기본으로 한다. 현재 값은 슬라이더 가까이에 명확히 표시한다.
- 선택 항목이 4개 이하이면 라디오 버튼을 사용하고, 5개 이상이면 콤보박스(셀렉트)를 사용한다.
- 모든 도구 화면은 모바일과 PC에서 같은 기능을 제공해야 한다. 모바일에서는 한 열 중심의 넉넉한 터치 영역을 사용하고, PC에서는 내용 폭과 입력 영역을 읽기 좋게 제한한다.
- Blogger 테마 원본은 `toolbox/theme/`에 두며 업로드용 XML과 편집용 CSS를 함께 관리한다.
- 업로드용 Blogger XML에는 소개글, 도구명, 버전, 카드, 검색 문구, 색상, 글꼴, 레이아웃을 직접 하드코딩하지 않는다. XML은 Blogger 기본 골조, 표시 위치, 외부 로더 주소를 담은 테마 변수만 제공한다.
- 사이트 디자인은 `assets/blogger/theme.css`, 동작은 `assets/blogger/site.js`, 소개와 문구는 `outputs/site.json`, 최신 도구 목록은 `outputs/tools.json`을 단일 원본으로 사용한다.
- 모든 도구 게시글은 `assets/blogger/theme.css`의 `.tb-post` 공통 테마와 `toolbox/templates/blogger-inline-tool-template.md`의 구조를 사용한다. 게시글 HTML에 공통 CSS를 반복하거나 `<style>` 블록을 넣지 않는다.
- 새 기능이나 게시글 수정 시 `.tb-hero`, `.tb-label`, `.tb-lead`, `.tb-features`, `.tb-frame`, `.tb-fallback`, `.tb-steps` 공통 클래스를 우선 재사용하고, 전역 외형 변경은 `assets/blogger/theme.css`에서 한 번만 처리한다.
- 페이지 로더는 `outputs/site.json`과 `outputs/tools.json`을 캐시 없이 다시 요청해 배포된 최신 값을 표시한다.
- Blogger 홈 화면은 소개와 검색 가능한 JSON 도구 목록을 먼저 표시하고 기존 게시글 목록도 유지한다. 도구 카드는 `postUrl`이 있을 때만 해당 Blogger 게시글로 이동한다.
- 모든 Blogger 게시글 페이지에는 블로그 메인의 도구 검색 영역으로 돌아가는 링크를 공통으로 표시한다.
- 공개 도구 게시글 제목은 `[초간단 툴박스] 도구명` 형식을 유지한다. 메인과 게시글 상단은 `outputs/site.json`의 Openworld Blog 소개와 동일한 도구 검색 UI를 공통으로 사용하며 Blogger 기본 헤더는 게시글에서 숨긴다.
- 도구 검색 결과는 검색어가 비어 있을 때 표시하지 않는다. 입력 뒤에만 `outputs/tools.json`의 제목, 설명, 해시태그와 20~30개 검색 키워드를 기준으로 일치·유사 결과를 가벼운 링크 목록으로 제공한다.
- 모든 공개 도구 게시글은 `toolbox/post-tags.json`에서 공통·도구별 해시태그를 생성해 본문에 표시하고 같은 값을 Blogger 라벨로 등록한다. 태그를 게시글 HTML이나 발행 스크립트에 개별 하드코딩하지 않는다.
- 한글 기본 글꼴은 Pretendard Variable 동적 서브셋을 사용하고, 로딩 실패 시 시스템 한글 글꼴로 폴백한다.
- 화면은 기능 구현 여부만이 아니라 사용자가 다음 행동을 쉽게 이해하고 실수를 바로 고칠 수 있는 UX를 기준으로 설계한다. 입력 부담, 인지 부하, 피드백 속도, 접근성, 모바일 터치 동선을 함께 검토한다.
- 비교 원문과 수정문처럼 각각 긴 내용을 입력하는 텍스트 영역이 둘 이상이면 화면 폭과 관계없이 위/아래 한 열로 고정한다. 각 입력 영역은 전체 가용 폭과 충분한 세로 높이를 사용하며, 모바일 세로 화면에서 가로 분할로 입력 폭을 줄이지 않는다.
- 참가자, 후보, 결과, 태그처럼 개수가 달라지는 다중 항목은 줄바꿈 텍스트만을 기본 입력 방식으로 사용하지 않는다. 번호가 있는 개별 입력 행, 명확한 `+ 항목 추가`, 행별 삭제, Enter로 다음 행 추가를 우선 제공한다.
- 다중 항목의 빈 행은 조용히 무시하지 않고 해당 행을 시각적으로 표시하며, 무엇을 입력해야 하는지 가까운 위치에 짧고 세련된 안내를 제공한다.
- 서로 한 쌍을 이루는 다중 값은 별도 목록으로 나누기보다 같은 행에 배치해 개수 불일치가 발생하지 않게 한다.
- 모든 공개 도구는 화면 상단에 공통 `KOR / ENG` 언어 선택기를 제공한다. 선택한 언어는 브라우저에 저장해 다른 도구에서도 이어서 사용하며, 기본값은 한국어다.
- 제목·라벨·버튼·힌트·오류·결과 상태·동적으로 추가되는 목록 행·접근성 문구까지 선택 언어로 표시한다.
- 언어 선택 UI와 상태 저장은 공통 모듈을 단일 원본으로 사용한다. 새 도구나 기존 도구 수정 시 개별 구현을 복제하지 않고 공통 모듈과 번역 사전을 연결한다.
- 결과가 무작위로 정해지는 놀이 도구는 `결과 코드 복사`, `코드로 결과 열기`, `PNG 저장`을 공통으로 제공한다.
- 결과 코드는 계정이나 서버 없이 같은 결과를 복원할 수 있어야 하며, 사용자에게 시드·직렬화 같은 기술 용어를 노출하지 않는다. 다른 도구의 코드나 손상된 코드는 쉬운 문장으로 입력 위치 가까이에서 안내한다.
- 복사 기능은 iframe의 클립보드 권한 차단을 정상 상황으로 처리한다. 최신 API 실패 시 호환 복사를 시도하고, 자동 복사가 모두 실패하면 오류를 노출하지 않고 직접 선택해 복사할 내용을 화면에 제공한다.
- 모바일 터치 환경에서는 초기 로드, 초기화, 항목 추가, 오류 안내, 복사 대체 처리 후 입력 요소에 자동 포커스하거나 자동 선택하지 않는다. 모든 도구는 `assets/toolbox-ux.js`의 공통 안전 포커스 헬퍼를 사용하고 직접 `focus()`를 호출하지 않는다.
- 입력·결과 구조와 사용 맥락이 사실상 같은 변환 기능은 형식 선택으로 한 도구에 통합한다. 통합된 기존 도구는 `versions.json`의 `status: retired`로 공개 카탈로그에서 제외하되, 기존 게시글 주소는 통합 도구 안내 또는 호환 진입점으로 유지한다.

## 의사결정 기록 및 업데이트

- 사용자가 전달한 기능 추가·수정 지시는 구현 전에 `toolbox/requests/*.md`와 관련 `toolbox/tools/*.md`에 먼저 정리한다.
- 결과가 달라지는 미확정 변환 규칙이나 공개 범위가 있으면 구현 전에 질의하고, 답변으로 확정된 내용만 Markdown에 기록한다.
- 요구사항, 구현 방향, 범위 변경이 확정되면 이 문서를 즉시 갱신한다.
- 아직 확정되지 않은 내용은 사실처럼 기록하지 않고 `결정 대기` 항목으로 남긴다.
- 기능 구현 전에 이 문서를 확인하고, 기존 원칙과 충돌하면 사용자에게 선택지를 제시한다.

## 첫 구현: QR · 바코드 생성기

- 첫 공개 도구는 QR 코드와 Code 128 바코드를 생성하는 단일 페이지 도구로 시작한다.
- 입력값은 텍스트 또는 URL이며, 생성 대상은 QR 코드와 바코드 중 하나를 라디오 버튼으로 선택한다.
- QR 코드 스타일은 기본 사각형과 둥근 모듈 스타일 두 가지를 라디오 버튼으로 제공한다. 둥근형은 세 위치 표시 마크도 둥근 모서리로 처리한다.
- 크기는 가로 슬라이더로 조절한다.
- 결과물은 화면에서 즉시 확인하고 PNG 또는 SVG로 저장할 수 있어야 한다.
- 블로그 본문에 넣을 수 있도록 도구 전용 임베드 URL과 복사 가능한 iframe 코드를 제공한다.
