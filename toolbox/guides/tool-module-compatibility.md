# 도구 모듈 호환성 가이드

## 적용 범위

이 가이드는 새로 만드는 기능 모듈과 새로 통합·분리하는 기능부터 적용한다. 기존 프로젝트 전체를 소급 재작성하지 않는다.

## 생성 시점의 기본 구조

1. 사용자가 여는 주소는 `embed/<tool>/index.html`로 고정한다.
2. 세부 기능은 독립 실행 가능한 모듈로 만들고 등록 JSON에서 연결한다.
3. `tool`, `module id`, 표시 제목, 카테고리와 결과 코드 형식을 서로 다른 값으로 관리한다.
4. 컨테이너는 모드 선택, 높이 조정, 공통 상태만 담당하고 기능 계산과 결과 생성은 모듈이 담당한다.
5. 모듈은 부모 DOM이나 Blogger 구조를 직접 참조하지 않는다.

## 모듈 등록 계약

각 항목은 다음 필드를 사용한다.

- `id`: 바뀌지 않는 모듈 식별자
- `label.ko`, `label.en`: 사용자 표시명
- `entry`: 통합 진입점 기준 실행 HTML 경로 또는 HTTPS 주소
- `messageTool`: 높이 메시지의 기존 도구 식별자
- `capabilities`: `resultCode`, `png`, `copyFallback`, `i18n` 지원 여부

새 필드는 추가할 수 있지만 기존 필드의 의미를 바꾸지 않는다. 표시명 변경 때문에 `id`, `entry`, 결과 코드 도구 식별자를 바꾸지 않는다.

## 통합과 은퇴

- 기존 도구를 통합할 때 버전·index·URL·공유 코드를 삭제하거나 재배정하지 않는다.
- 은퇴 도구는 `status: retired`와 `replacementTool`로 공개 카탈로그에서만 제외한다.
- 기존 모듈이 계약을 모두 구현하지 않아도 등록 어댑터로 연결할 수 있다. 새 모듈은 생성 시점부터 계약을 직접 구현한다.
- 통합 컨테이너가 없어도 모듈 단독 주소에서 핵심 기능이 동작해야 한다.

## 검증

- `node scripts/validate-tool-modules.mjs`로 등록 필드, 중복 id, 실행 경로와 기능 지원 값을 확인한다.
- 컨테이너와 단독 모듈에서 PC·모바일 높이, 가로 넘침, 언어, 오류와 결과 복원을 각각 확인한다.
